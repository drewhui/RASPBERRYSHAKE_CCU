這個網站背後是一套跑在 Raspberry Pi 4 上的簡化版地震早期預警（EEW）系統， 核心是 Earthworm 8.0——地震學界行之有年的開源即時地震資料處理框架，用 Docker 容器化部署 （ARM64）。系統即時接收 RaspberryShake RE81D 地震儀的資料，同時跑三層告警 （單站 onsite 快報、測站即時震度、多站協同的網路型 EEW 解算）。 這個網站顯示的 PGA / PGV / 波形，是這套系統其中一條資料支線的公開 輸出，不是獨立產品。完整版見 ARCHITECTURE.md。

資料來源

  • RaspberryShake RE81D（RS1D 型號，單一垂直分量檢波器，100 sps）：跟 Pi 在同一個區域網路，透過 SeedLink 協定即時串流波形——這是本站顯示資料的來源。
  • IRIS 遠端測站（中央氣象署寬頻地動觀測網 TW 網 + IU 網 TATO 站）：透過網際網路連到 IRIS 的 SeedLink 伺服器拉取即時資料，提供多站定位 / 規模估算所需的額外測站，不是本站顯示的對象，但共用同一套 Earthworm 處理鏈。

Earthworm 核心架構

Earthworm 用「共享記憶體環（ring）+ 多個獨立小程式（module）」的模型運作：每個 module 是獨立 process，讀寫特定的 ring，靠 ring 裡不同類型的訊息（波形、pick、事件⋯）溝通，彼此不直接呼叫。目前跑的 module（依啟動順序）：

Module角色
statmgr系統心跳 / 健康監控
slink2ew ×2SeedLink 客戶端，一個接 IRIS 遠端測站、一個接本地 RaspberryShake，寫入 WAVE_RING
pick_ew波形挑波（picking）：STA/LTA 演算法即時偵測 P 波到時，同時算出 onsite 快報用的 Pa/Pv/Pd/τc，寫入 PICK_RING
tcpd多站協同（coincidence）關聯：比對多測站 P 波 pick，達門檻站數才組成事件解，估算震央與規模（Mpd），寫入 EEW_RING
dcsn_xml把 tcpd 的事件解格式化成 EEW 警報 XML
wave_serverV波形環狀歸檔，供外部工具（如 Swarm）查詢 / 繪圖
diskmgr / copystatus磁碟空間管理、狀態訊息轉發到 HYPO_RING

三層告警（由快到慢，由單站到多站）

  1. Onsite τc-Pd 快報——pick_ew 每完成一個 P 波 pick 就順便算出 Pa/Pv/Pd/τc，套用 Wu & Kanamori 準則（Pd、τc 同時超過門檻）判斷是否可能是具破壞力的地震。單一測站、P 波到站後數秒內就有結果，是最快但誤報率也最高的一層。
  2. 測站即時震度——自訂的 station_watch.py（不是 Earthworm 內建，用 PyEW 直接讀 WAVE_RING）即時把原始計數值換算成地動速度，量測 PGV，再微分成 PGA，對照中央氣象署 2020 震度分級表。這是「量出來的」不是預測，不會錯，只受限於資料延遲（本地 RaspberryShake 約 1–2 秒，IRIS 遠端站約 10–20 秒）。
  3. 網路型 EEW——tcpd 需要多個測站的 P 波 pick 互相吻合（到時、距離都在門檻內）才會組成一組解，dcsn_xml 再轉成正式 XML 警報。這層最穩定（多站交叉驗證），但也最慢，且需要數個測站同時觸發才能運作——單靠這一台 RaspberryShake 無法獨立完成，解算結果不在這個網站顯示範圍內。

已知限制

  • RaspberryShake RE81D 是 RS1D 型號，硬體上只有一個垂直分量檢波器，沒有水平分量、沒有強震型加速度計——4 級以上的量測可能因訊號削波而失真。
  • 測站位置只標示大略城市，不公開精確座標。
  • GitHub Pages 的 CDN 對資料檔另有最長 10 分鐘快取，實際顯示會落後現況約 1–11 分鐘。

This site is the public face of a simplified earthquake early warning (EEW) system running on a Raspberry Pi 4, built around Earthworm 8.0 — a long-established open-source real-time seismic data processing framework in the seismology community, deployed in Docker (ARM64). The system ingests live data from a RaspberryShake RE81D seismometer and runs three alert tiers concurrently (single-station onsite alerting, per-station real-time intensity, and multi-station network EEW solutions). The PGA / PGV / waveform data shown on this site is the public output of just one branch of that pipeline, not a standalone product. Full write-up: ARCHITECTURE.md.

Data Sources

  • RaspberryShake RE81D (RS1D model, single vertical-component geophone, 100 sps): on the same local network as the Pi, streamed live over the SeedLink protocol — this is the source of the data shown on this site.
  • Remote IRIS stations (Taiwan CWA broadband network TW + IU network's TATO station): pulled live over the internet from IRIS's SeedLink server, providing the extra stations needed for multi-station location / magnitude estimation. Not shown on this site, but processed through the same Earthworm pipeline.

Earthworm Core Architecture

Earthworm runs on a "shared-memory ring + independent small programs (modules)" model: each module is its own process, reading and writing specific rings, communicating via typed messages on the ring (waveform, pick, event…) rather than calling each other directly. Modules currently running (in startup order):

ModuleRole
statmgrSystem heartbeat / health monitoring
slink2ew ×2SeedLink clients — one for the remote IRIS stations, one for the local RaspberryShake — writing into WAVE_RING
pick_ewWaveform picking: an STA/LTA algorithm detects P-wave arrivals in real time, also computing Pa/Pv/Pd/τc for onsite alerting, writing into PICK_RING
tcpdMulti-station coincidence association: correlates P-wave picks across stations, forms an event solution once enough stations agree, estimates epicenter and magnitude (Mpd), writing into EEW_RING
dcsn_xmlFormats tcpd's event solutions into EEW alert XML
wave_serverVRing-buffer waveform archive, queryable by external tools (e.g. Swarm)
diskmgr / copystatusDisk space management, status message relay to HYPO_RING

Three Alert Tiers (fast to slow, single-station to multi-station)

  1. Onsite τc-Pd alert — every time pick_ew completes a P-wave pick, it also computes Pa/Pv/Pd/τc, applying the Wu & Kanamori criterion (Pd and τc both above threshold) to judge whether the event may be damaging. Single-station, results within seconds of the P wave arriving — the fastest tier, but also the one with the highest false-alarm rate.
  2. Per-station real-time intensity — a custom station_watch.py (not built into Earthworm, reads WAVE_RING directly via PyEW) converts raw counts to ground velocity in real time, measuring PGV, differentiating it into PGA, and mapping against Taiwan CWA's 2020 intensity scale. This is a direct measurement, not a prediction — it can't be "wrong" in that sense, only limited by data latency (~1–2s for the local RaspberryShake, ~10–20s for the remote IRIS stations).
  3. Network EEW — tcpd requires P-wave picks from multiple stations to be mutually consistent (arrival time and distance both within threshold) before forming a solution; dcsn_xml then converts it into a formal XML alert. This is the most stable tier (cross-validated across stations), but also the slowest, and needs several stations to trigger together — a single RaspberryShake can't produce this on its own, and its output isn't shown on this site.

Known Limitations

  • The RaspberryShake RE81D is an RS1D model — hardware-wise it has only a single vertical-component geophone, no horizontal components, and no strong-motion accelerometer, so readings above roughly intensity level 4 may be distorted by signal clipping.
  • Station location is shown at city-level only — precise coordinates aren't published.
  • GitHub Pages' CDN caches the data files for up to 10 minutes on top of the push interval, so the displayed data can lag reality by roughly 1–11 minutes.
